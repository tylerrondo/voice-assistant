# Development Workflow

## Назначение
Описывает стандартный процесс разработки и проверки изменений в платформе.

## Принципы
- Новый канал никогда не изменяет Platform Core
- Каждое изменение проходит через Validation Bench
- Без проверки сценариев функциональность считается неподтверждённой

## Процесс разработки

### 1. Изучить документацию
Перед началом работы прочитать:
- docs/ARCHITECTURE.md
- docs/PLATFORM_RULES.md
- docs/MIGRATING_NEW_CHANNEL.md

### 2. Создать компонент
Реализовать только необходимое:
- Provider
- Mapper
- Channel

Не трогать Platform Core без крайней необходимости.

### 3. Подключить к Emulator
Проверить работу без реального приложения.

### 4. Создать сценарии
Добавить сценарии в Scenario Engine.

### 5. Проверить Execution Log
Убедиться что все действия записываются корректно.

### 6. Запустить Verification Harness
Убедиться что все проверки проходят (PASS).

### 7. Проверить через Validation Bench
Открыть стенд на Vercel.
Проверить глазами и на слух.
Сформировать отчёт.

### 8. Отправить отчёт
Нажать Send Report.
Отчёт отправляется через /api/v1/mail/.

### 9. Интегрировать в приложение
Только после успешного прохождения всех этапов.

## Правила PR
- Один PR — одна задача
- Каждый PR сопровождается обновлением документации
- История PR фиксируется в docs/history/
- RFC документы хранятся в docs/rfc/

## Запрещено
- Менять Platform Core без подтверждения нескольких каналов
- Создавать дублирующую инфраструктуру
- Интегрировать без прохождения Validation Bench
- Создавать собственный SMTP или авторизацию
